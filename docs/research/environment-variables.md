# Environment Variables

Research into every environment variable referenced in the README, verifying whether each one actually affects program behavior.

## Summary

All proxy-side environment variables (`MODEL_FALLBACK_*`, `GH_TOKEN`, `HTTP_PROXY`/`HTTPS_PROXY`) are functional, though some have caveats about scope. One widely-circulated Claude Code variable (`DISABLE_NON_ESSENTIAL_MODEL_CALLS`) turned out to be fake and was removed. Another (`ANTHROPIC_SMALL_FAST_MODEL`) is deprecated and was also removed from the `--claude-code` output.

## Proxy-Side Variables

### `MODEL_FALLBACK_CLAUDE_OPUS` / `MODEL_FALLBACK_CLAUDE_SONNET` / `MODEL_FALLBACK_CLAUDE_HAIKU`

**Status:** Functional, but only on the chat completions translation path.

These variables override the default model used when the requested model ID is not in Copilot's known model list and starts with `claude-opus-`, `claude-sonnet-`, or `claude-haiku-`.

**Call chain:**

```text
process.env.MODEL_FALLBACK_CLAUDE_*
  → getModelFallbackConfig()                    [src/lib/model-resolver.ts]
    → createAnthropicAdapter()                  [src/routes/messages/strategies/shared.ts]
      → AnthropicMessagesAdapter.toCapiPlan()   [src/adapters/anthropic-messages-adapter.ts]
        → buildCapiExecutionPlan()              [src/core/capi/plan-builder.ts]
          → plan.payload.model = resolvedModel  ← sent to Copilot API
```

**Priority:** env var > `config.json` `modelFallback.*` > built-in default.

**Caveat:** `resolveModel()` is only called from `createAnthropicAdapter()`, which is only used by the **chat completions strategy** and **count_tokens handler**. The other two `/v1/messages` strategies bypass model resolution entirely:

| Execution path | Model resolution applied |
|----------------|------------------------|
| Chat Completions strategy | Yes |
| Token counting (`/v1/messages/count_tokens`) | Yes |
| Native Messages strategy | No — model passed through as-is |
| Responses API strategy | No — model passed through as-is |

### `GH_TOKEN`

**Status:** Functional. Docker-only entry point.

Used exclusively in `entrypoint.sh` to pass a GitHub token to the `--github-token` CLI flag:

```text
$GH_TOKEN (Docker env)
  → entrypoint.sh: -g "$GH_TOKEN"
    → start.ts: options.githubToken
      → state.auth.githubToken
        → GitHub API: authorization: token <github_token>
          → GET /copilot_internal/v2/token
            → state.auth.copilotToken
              → All Copilot API requests: Authorization: Bearer <copilot_token>
```

The GitHub token is not used directly for Copilot API calls. It is exchanged for a short-lived Copilot token via GitHub's internal API, and that Copilot token is what authenticates all upstream requests.

**Empty string handling:** When `GH_TOKEN` is not set in Docker, `entrypoint.sh` passes `-g ""`. The JavaScript falsy check `if (options.githubToken)` treats empty string as "not provided", so the interactive OAuth device-code flow is triggered instead. This is correct behavior.

### `HTTP_PROXY` / `HTTPS_PROXY`

**Status:** Functional on both runtimes, but via different mechanisms.

The `--proxy-env` CLI flag calls `initProxyFromEnv()` in `src/lib/proxy.ts`, which configures an `undici` global dispatcher that routes requests through the proxy URL returned by the `proxy-from-env` library. However, this function has an early return on Bun:

```text
if (typeof Bun !== 'undefined')
  return // no-op on Bun
```

This means the `--proxy-env` flag's code path is **only effective on Node.js**.

On **Bun** (the recommended runtime), proxy support works anyway because Bun's native `fetch` implementation reads `HTTP_PROXY`/`HTTPS_PROXY` from `process.env` automatically. The `--proxy-env` flag is silently ignored but proxying still works.

| Runtime | `--proxy-env` code runs | Proxy works | Mechanism |
|---------|------------------------|-------------|-----------|
| Bun | No (early return) | Yes | Bun native `fetch` reads env |
| Node.js | Yes | Yes | `undici` global dispatcher via `proxy-from-env` |

## Claude Code Client Variables

These variables are not consumed by ghc-proxy itself. They are either generated by the `--claude-code` mode for clipboard output, or documented in the README's `settings.json` example for users to configure Claude Code.

### Verified as real Claude Code variables

Source: official docs at `code.claude.com/docs/en/settings` and grep of `@anthropic-ai/claude-code@2.1.74` bundle (`cli.js`).

| Variable | Purpose | In Claude Code bundle |
|----------|---------|----------------------|
| `ANTHROPIC_BASE_URL` | API endpoint override | Yes |
| `ANTHROPIC_AUTH_TOKEN` | API authentication | Yes |
| `ANTHROPIC_MODEL` | Primary model selection | Yes |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | Sonnet-tier model | Yes |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | Haiku-tier (fast/cheap) model | Yes |
| `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` | Disables telemetry, error reporting, autoupdater, bug command | Yes |

### `DISABLE_NON_ESSENTIAL_MODEL_CALLS` — Not real

**Status:** Removed. Never existed in Claude Code.

This variable appeared in the README's `settings.json` example and was generated by `--claude-code` mode. Investigation found:

1. **Not in official docs** — `code.claude.com/docs/en/settings` does not list it
2. **Not in Claude Code source** — grep of the `cli.js` bundle (v2.1.74) for all `DISABLE_*` variables found 40+ entries; none matching `DISABLE_NON_ESSENTIAL_MODEL_CALLS` or any `*MODEL_CALL*` pattern
3. **Not in CHANGELOG** — the `CHANGELOG.md` in the `anthropics/claude-code` repo has no mention
4. **No code path reads it** — setting this variable has zero effect on Claude Code behavior

The variable is widely copied across blog posts and configuration guides on the internet, likely originating from an early tutorial's guess at the variable name. The real variable that controls non-essential traffic is `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` (note: `NONESSENTIAL` as one word, not `NON_ESSENTIAL`).

### `ANTHROPIC_SMALL_FAST_MODEL` — Deprecated

**Status:** Removed from `--claude-code` output. Deprecated in favor of `ANTHROPIC_DEFAULT_HAIKU_MODEL`.

The official docs page lists `ANTHROPIC_SMALL_FAST_MODEL` with a "(deprecated)" annotation. `ANTHROPIC_DEFAULT_HAIKU_MODEL` serves the same purpose and is the current recommended variable. The `--claude-code` codegen was setting both to the same value, which was redundant.

## Changes Made

Based on this research, the following corrections were applied:

### `src/start.ts`
- Removed `ANTHROPIC_SMALL_FAST_MODEL` from `--claude-code` env generation (deprecated, redundant with `ANTHROPIC_DEFAULT_HAIKU_MODEL`)
- Removed `DISABLE_NON_ESSENTIAL_MODEL_CALLS` from `--claude-code` env generation (fake variable)

### `README.md`
- Removed `DISABLE_NON_ESSENTIAL_MODEL_CALLS` from `settings.json` example and variable table
- Updated `claude-sonnet` default fallback from `4.5` to `4.6` (three occurrences: default table, env example, config.json example)
- Clarified `--proxy-env` description: Node.js only; Bun reads proxy env natively
- Added note that model fallbacks only apply to the chat completions translation path
